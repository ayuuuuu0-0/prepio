## Project Overview

Prepio is a gamified interview preparation platform. The experience mirrors
Duolingo: daily streaks, gem rewards, unlockable animal characters, competitive
notifications, and weekend deep-dive content. The stack is:

- **Backend:** Go (REST + WebSocket, event-driven)
- **Web Frontend:** Next.js (App Router)
- **Mobile:** Flutter (Android + iOS, single codebase)
- **Database:** PostgreSQL (primary), Redis (streaks, leaderboards, sessions)
- **Queue:** Kafka (event bus between services)
- **Infra:** Docker, deployable to any VPS or cloud

Read `ARCHITECTURE.md` before touching any service. Every decision about
data ownership, API contracts, and service boundaries has already been made.
Your job is to implement, not redesign.

## The Non-Negotiables

Hard rules. Do not break them. Do not ask if they apply to your case. They always apply.

**Never own streak logic outside the streak service.**
One service computes, stores, and updates streaks. No other service, handler,
or Flutter widget calculates whether a streak is alive or broken. If you are
writing streak logic anywhere outside `services/streak/`, you are wrong.

**Never push notifications directly from a feature handler.**
All notifications go through the notification service via the Kafka topic
`notifications.dispatch`. A handler that calls FCM or APNs directly is wrong.
Produce an event, let the notification service consume it.

**Never hardcode a reward value, gem cost, or XP amount.**
Every number that affects progression lives in `config/rewards.go`. Gem costs,
XP per question, streak freeze price, character unlock threshold вҖ” all of it.
If you are typing `50` or `100` as a raw literal in a handler or Flutter widget,
move it to config first.

**Never write question content inline in code.**
Questions, answers, hints, and company tags live in the database, seeded via
migration files. No question string appears in Go structs, Flutter widgets, or
Next.js components. If you need a question in a test, use the test fixture
factory in `test/factories/question.go`.

**Never mutate user progress state from more than one service.**
The progress service owns XP, level, and gem balance. The streak service owns
streak count and freeze state. No other service writes to these tables. Other
services produce Kafka events; the owning service consumes and applies them.

**Never skip the review queue for AI-generated questions.**
Questions generated by the AI content pipeline are written to
`questions.pending` status. They go live only after a human review sets status
to `approved`. No code path promotes a question from pending to live
automatically.

**API Gateway** authenticates requests and routes them. It never contains
business logic. No streak checks, no gem deduction, no question selection.

**User Service** owns the users table, auth tokens, profile, and character
unlocks. It does not know what a streak is.

**Question Service** owns the question bank, company tags, difficulty levels,
daily paper generation, and answer submission evaluation. It produces a
`question.answered` event to Kafka on every submission. It does not award XP.

**Streak Service** consumes `question.answered` events. It decides if today's
activity qualifies, updates streak count, handles streak freezes, and produces
`streak.updated` events.

**Progress Service** consumes `streak.updated` and `question.answered` events.
It awards XP and gems, handles level-ups, and produces `progress.updated`
events.

**Notification Service** consumes all events and decides what to send, when,
and to whom. It owns all FCM/APNs calls.

## Streak Rules (Source of Truth)

These rules are implemented once, in the streak service. Everywhere else that
needs to display streak data reads from the streak service API.

- A streak increments when a user answers at least one question in a calendar
  day in their local timezone. Timezone is stored on the user record.
- A streak is broken if no qualifying answer is submitted by 23:59 in the
  user's local timezone.
- A streak freeze costs `config.StreakFreezeGemCost` gems. It prevents one
  missed day from breaking the streak. It is consumed automatically on the
  first missed day. A user can hold at most `config.MaxStreakFreezes` freezes.
- Streaks are stored in Redis with a TTL of 36 hours. PostgreSQL is the
  source of truth. Redis is a cache, not the record of authority.

## Gamification System

**Gems**
Earned by answering questions, maintaining streaks, completing weekly
challenges. Spent on streak freezes and character unlocks. Gem balance lives
in the progress service. Gem costs live in `config/rewards.go`.

**XP and Levels**
XP is awarded per question answered, with multipliers for difficulty and
company tier. Level thresholds live in `config/levels.go` as a slice.
Level-up triggers a `progress.levelup` Kafka event which the notification
service uses to send a push.

**Characters**
Animal-based characters. Each has a name, species, unlock cost in gems, and
a set of notification dialogue lines stored in the database. The default
character is free. Characters are unlocked via the user service after the
progress service confirms sufficient gem balance via a synchronous internal
API call (not Kafka, because this is a transactional operation requiring
immediate consistency).

**Notification Dialogue**
Each character has 5 to 10 notification lines per notification type
(streak reminder, streak broken, level up, weekly challenge). The notification
service selects one randomly per send. Dialogue lines live in the database
under `character_dialogues`, never hardcoded in Go or Dart.

**Weekly Challenge**
Every Saturday and Sunday, the question service serves a different content
format: a full system design problem or a multi-part DSA set. Completion
awards a `config.WeekendChallengeGemBonus` gem bonus. The question service
determines weekend vs weekday content based on the user's local day of week.

**Leagues and Leaderboards**
Weekly leaderboard resets every Monday 00:00 UTC. Ranking is by XP earned in
the current week. Leaderboard data lives in Redis sorted sets, backed by
PostgreSQL for persistence. The leaderboard is scoped to users at similar
XP levels (within one level band) to keep competition meaningful.

## Notification Rules

All notifications are competitive and specific, never generic.

**Streak reminder (sent at user's configured reminder time, default 21:50):**
Uses the user's active character's dialogue. Example tone: "Ayush hasn't
practiced today. 47 people in your league already have."

**Streak broken:**
Sent at 00:05 if no activity that day. One notification, never repeated.

**Level up:**
Sent immediately on level-up event.

**League position change:**
Sent when a user moves up or down 3 or more positions in their league.
Batched hourly, not real-time, to avoid spam.

**Never send more than 3 notifications in a 24-hour window per user.**
The notification service enforces this limit using a Redis counter with a
24-hour TTL keyed by user ID.

## API Design Rules

**REST for CRUD and queries. WebSocket for real-time leaderboard updates only.**

**Every endpoint is versioned under `/api/v1/`.**
Do not create unversioned endpoints.

**Request and response structs are defined in `internal/dto/`.**
Handlers do not construct raw `map[string]any` responses. They return a typed
DTO that the response encoder serializes.

**Errors follow a consistent envelope:**
```json
{
  "error": {
    "code": "streak_freeze_insufficient_gems",
    "message": "not enough gems to purchase a streak freeze"
  }
}
```
Error codes are constants defined in `constants/errors.go`. Never return a
raw error string from `fmt.Errorf` directly to the client.

**Pagination is cursor-based, never offset-based.**
Leaderboards and question history use a `cursor` and `limit` query param.
Offset pagination breaks under concurrent writes. Do not use it.

## Flutter App Rules

**State management: Riverpod only.**
Do not use Provider, BLoC, GetX, or setState outside of truly local ephemeral
UI state (e.g. a text field focus). All server state, streak data, gem balance,
and question state go through Riverpod providers.

**Never call the API directly from a widget.**
Widgets talk to providers. Providers talk to repositories. Repositories talk
to the API client. A widget that imports `http` or `dio` directly is wrong.

**The streak display widget reads from a single `StreakProvider`.**
No other widget computes whether a streak is active. They all read the same
provider.

**Notifications are handled by `flutter_local_notifications` + FCM.**
The app registers the FCM token on login and sends it to the user service.
The notification service stores it and uses it for all pushes.

**Offline support for question answering:**
Answers submitted while offline are queued locally in Hive and synced when
connectivity resumes. The question service accepts out-of-order submissions
with a `submitted_at` timestamp and evaluates streak eligibility by that
timestamp, not server receipt time.

**Never hardcode any string visible to the user.**
All user-facing strings live in `lib/l10n/` ARB files, even if the app
launches English-only. Internationalisation is easier to add than to retrofit.

## Database Rules

**PostgreSQL owns all persistent state. Redis owns all ephemeral or
cache state.**

Tables that Redis caches must have PostgreSQL as the write-through source.
Never write to Redis without also writing to PostgreSQL in the same
transaction or immediately after.

**Every table has `created_at` and `updated_at` columns.**
Set them via a trigger, not application code.

**Migrations live in `migrations/` and are numbered sequentially.**
`001_create_users.sql`, `002_create_questions.sql`. Never edit a migration
that has already been applied to production. Write a new one.

**No raw SQL in handlers or services.**
SQL lives in `internal/store/` query files. Services call store methods.
If you are writing a SQL string inside a handler, move it.

## Content Pipeline Rules

The question bank is the product's moat. Treat it accordingly.

**Question fields:**
- `body` вҖ” the question text
- `company_tags` вҖ” array of company names (Google, Amazon, Zepto, etc.)
- `round_type` вҖ” enum: `dsa`, `system_design`, `lld`, `aptitude`, `fundamentals`, `behavioral`
- `difficulty` вҖ” enum: `easy`, `medium`, `hard`
- `answer_guide` вҖ” the evaluator's rubric, not shown to the user
- `status` вҖ” enum: `pending`, `approved`, `retired`

**AI-generated questions use the content pipeline service.**
The pipeline calls the LLM, formats the response into the question schema,
writes to `questions` with `status = pending`, and notifies the review queue.
It never promotes questions itself.

**Question selection for daily paper:**
The question service selects one question per session using this priority:
1. Questions the user has never seen from their target companies
2. Questions from their target companies at their current difficulty band
3. Any unseen question at their difficulty band
4. Fallback: a random approved question

The minimum to count toward a streak is one submitted answer. The daily
paper can contain up to `config.DailyPaperMaxQuestions` questions but only
one is required.

## Go-Specific Rules

**Do not use `init()` functions.**
Explicit setup in `main.go` only.

**Do not use `panic()` in service, handler, or store code.**
Return errors. Handle them. Panic only for genuinely unrecoverable programmer
errors like a nil required dependency at startup.

**Always pass `context.Context` as the first argument to any function that
does I/O, calls another service, or runs in a goroutine.**

**Use `time.NewTicker()` for any periodic work.**
`time.Sleep()` drifts. Do not use it for scheduling.

**Channel directions must be typed in every function signature.**
`ch <-chan Event` for read-only. `ch chan<- Event` for write-only.

**Error strings are lowercase with no trailing punctuation.**

**Unexported by default. Export only what another package needs.**

**No `utils` or `helpers` packages.**
Code belongs to the package that owns its concern.

**Test factories produce realistic but fake data.**
Never use real user IDs, real company names as primary test identifiers, or
production database connections in tests.

**Every test that touches PostgreSQL or Redis uses a test container.**
`testcontainers-go` spins up isolated instances. Tests never share state.

**Streak and progress logic must have unit tests covering:**
- First question of the day increments streak
- Second question of the day does not double-increment
- Missed day without freeze breaks streak
- Missed day with freeze consumes freeze and holds streak
- Gem deduction on streak freeze purchase
- Character unlock fails when balance is insufficient
